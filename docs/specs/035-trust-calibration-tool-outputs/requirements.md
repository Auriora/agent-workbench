# Requirements

## Problem Context

Agent Workbench responses already expose freshness, analysis validity,
capability level, evidence kinds, verification status, caveats, warnings, and
errors. Those fields help agents route work, but they do not provide one
centralized contract that says what a response is safe to use for, what it is
not safe to use for, and what must be verified before an agent makes a stronger
claim.

The backlog records this as EB023: agents can over-read useful routing evidence
as proof unless major responses calibrate trust consistently. Existing durable
docs already state that routing evidence is not proof, planned validation is not
completed validation, and direct reads or executed validation are required for
stronger claims. This spec turns that prose rule into an additive runtime
contract and shared presenter behavior.

## Goals

- Add centralized trust calibration to Agent Workbench response metadata or a
  closely related shared response layer.
- Preserve existing capability, evidence, freshness, analysis-validity, and
  verification-status vocabulary.
- Make trust calibration generated by shared helpers and presenters rather than
  duplicated in individual MCP adapters.
- Cover the major tool families agents rely on for routing, planning,
  verification, and proof claims.
- Add fixture-backed tests proving routing, parser-backed, semantic,
  direct-read, planned-validation, executed-validation, and proof evidence stay
  distinct.

## Non-Goals

- Do not redesign the full response envelope or introduce a breaking contract
  version unless design proves the additive path is not viable.
- Do not add command execution or automatic validation.
- Do not claim vulnerability presence or absence, whole-program semantic
  correctness, or mutation safety from routing evidence.
- Do not duplicate long natural-language caveats in every adapter.
- Do not broaden parser, semantic, validation, or command-execution fallbacks.

## Durable Source Baseline

- `docs/reference/runtime-contracts.md` owns response metadata vocabulary,

- `docs/design/mcp-surface-design.md` owns MCP response behavior, routing

- `docs/reference/documentation-map.md` identifies runtime contracts as the

## Glossary

- **Trust calibration:** A compact, machine-readable summary of what a response
  can safely support, what it cannot support, and which evidence would be
  required for a stronger claim.
- **Routing evidence:** Evidence that helps an agent decide where to inspect
  next. It is not proof of behavior or safe mutation.
- **Claim-supporting evidence:** Executed validation, direct-read evidence, or
  fixture-backed semantic evidence that can support a bounded claim. Executed
  validation is required before claiming tests or checks passed; direct-read and
  semantic evidence remain scoped to the exact material and operation they
  cover.
- **Major tool family:** A response class used frequently by agents, including
  context routing, docs tools, graph tools, diagnostics, verification planning,
  workspace edit preview/apply, repository resources, and integration health.

## Requirement Details

### Requirement 1: Central Trust Calibration Contract

**User Story:** As an agent consuming Workbench responses, I want every major
response to state safe and unsafe uses, so that I do not turn routing evidence
into proof.

#### R1 Acceptance Criteria

1. WHERE any public MCP resource or tool returns a standard response envelope,
   THE SYSTEM SHALL include centralized trust calibration fields in the response
   metadata or a shared response-adjacent structure unless that surface has an
   explicit documented exclusion.
2. WHERE trust calibration is present, THE SYSTEM SHALL include
   `safe_to_use_for`, `not_safe_to_use_for`, and `must_verify_by` semantics.
3. WHERE existing response metadata already states freshness, capability,
   evidence kinds, analysis validity, verification status, caveats, warnings,
   or errors, THE SYSTEM SHALL preserve those fields and their current meanings.
4. IF a response has no useful evidence, stale evidence, partial evidence, or
   environment-blocked evidence, THEN THE SYSTEM SHALL calibrate the response as
   unsafe for proof or completion claims.
5. IF a response is routing-only, THEN THE SYSTEM SHALL state that it is safe
   for navigation or next-read selection and not safe for implementation,
   completion, closure, or proof without further evidence.
6. IF a response contains warnings, errors, invalid input, blocked validation,
   degraded runtime state, stale evidence, or environment failure metadata, THEN
   THE SYSTEM SHALL include trust calibration that names the blocked or unsafe
   use and the evidence required before relying on the result.
7. WHERE trust calibration is added to the response contract, THE SYSTEM SHALL
   use optional or otherwise backward-compatible fields unless a design-stage
   migration explicitly justifies a contract-version change.

### Requirement 2: Shared Generation Policy

**User Story:** As a maintainer, I want trust calibration generated from shared
metadata helpers, so that tool adapters do not drift or copy inconsistent
phrases.

#### R2 Acceptance Criteria

1. WHERE trust calibration is derived from existing metadata, THE SYSTEM SHALL
   derive it through shared helpers or shared presenter policy.
2. WHERE a tool family needs a specialized calibration, THE SYSTEM SHALL express
   that specialization as a typed policy input rather than hard-coded adapter
   prose.
3. IF two tool families share the same trust state, THEN THE SYSTEM SHALL
   produce equivalent safe/unsafe/verify semantics.
4. IF a new public MCP surface is registered without trust calibration or an
   explicit exclusion, THEN tests SHALL fail or flag the missing policy.
5. THE SYSTEM SHALL keep normal output compact and avoid repeated paragraph-size
   caveats when structured trust fields carry the same information.

### Requirement 3: Evidence Distinctions Stay Explicit

**User Story:** As an agent reporting results to a user, I want Workbench to
separate routing, parser, semantic, direct-read, planned-validation,
executed-validation, and bounded claim-supporting evidence, so that my final
answer does not overclaim.

#### R3 Acceptance Criteria

1. WHERE a response is backed by lexical, heuristic, docs search, or
   resource-backed evidence, THE SYSTEM SHALL classify it as routing evidence
   unless stronger evidence is explicitly present.
2. WHERE a response is parser-backed or partial-semantic, THE SYSTEM SHALL
   allow safe use for local structure and navigation while requiring
   verification for broad impact, mutation safety, or behavior claims.
3. WHERE a response is direct-read backed, THE SYSTEM SHALL allow safe use for
   precise claims about the read section or file while keeping broader claims
   scoped to the read evidence.
4. WHERE validation is planned but not executed, THE SYSTEM SHALL mark it as
   safe for validation planning and not safe for claiming tests/checks passed.
5. WHERE validation has executed or equivalent evidence is available, THE
   SYSTEM SHALL allow passed-validation or proof claims only for the bounded
   checks or evidence actually represented.

### Requirement 4: Major Tool Family Coverage

**User Story:** As a coding agent, I want consistent trust calibration across
the Workbench tools I use most, so that I do not have to infer safety rules
from tool-specific wording.

#### R4 Acceptance Criteria

1. WHERE `context_for_task` returns requested files, related files, governing
   docs, ranked symbols, validation hints, lifecycle evidence, risks, or next
   actions, THE SYSTEM SHALL calibrate those results as routing and planning
   evidence unless direct-read or executed-validation evidence is present.
2. WHERE docs search, docs overview, docs map, docs outline, or
   docs-current-for-task returns results, THE SYSTEM SHALL distinguish routing
   results from direct-read section evidence.
3. WHERE symbol search, references, or impact returns parser, graph, heuristic,
   or confidence evidence, THE SYSTEM SHALL calibrate the safe scope of that
   evidence and the required verification for mutation or blast-radius claims.
4. WHERE diagnostics or post-edit feedback returns findings, THE SYSTEM SHALL
   distinguish static diagnostics and planned validation from executed checks.
5. WHERE `verification_plan` returns commands, THE SYSTEM SHALL calibrate them
   as planned evidence unless an explicit executed-validation input exists.
6. WHERE preview/apply workspace edit surfaces return edit tokens or outcomes,
   THE SYSTEM SHALL calibrate preview evidence, applied edit evidence, and
   residual validation needs separately.
7. WHERE repository status, scope, overview, or integration health returns
   runtime health, THE SYSTEM SHALL calibrate those responses as runtime
   availability and routing evidence, not proof of requested task completion.

### Requirement 5: Contract And Golden Test Coverage

**User Story:** As a maintainer changing response presenters, I want tests that
fail on trust drift, so that confidence labels do not silently weaken over time.

#### R5 Acceptance Criteria

1. GIVEN the response metadata contract, WHEN trust calibration fields are
   added, THEN contract tests SHALL validate the schema and enum vocabulary.
2. GIVEN each public MCP surface covered by the implementation slice, WHEN
   golden responses are generated, THEN tests SHALL assert representative
   `safe_to_use_for`,
   `not_safe_to_use_for`, and `must_verify_by` semantics.
3. GIVEN routing-only evidence, WHEN tests inspect the response, THEN they SHALL
   fail if the response is labeled as proof.
4. GIVEN planned validation evidence, WHEN tests inspect the response, THEN they
   SHALL fail if the response is labeled as executed validation.
5. GIVEN direct-read or executed-validation evidence, WHEN tests inspect the
   response, THEN they SHALL verify the safe scope remains bounded to that
   evidence.
6. GIVEN an error, warning, blocked, invalid, stale, degraded, or
   invalid-due-to-environment response, WHEN tests inspect the response, THEN
   they SHALL verify trust calibration does not label the response as proof or
   completion evidence.
7. GIVEN an older consumer that ignores additive trust fields, WHEN the response
   is parsed through the existing envelope contract, THEN existing metadata
   fields SHALL retain their current meanings.

## Correctness Properties

- **CP001 Routing Is Not Proof:** For every response with only routing evidence,
  trust calibration must exclude proof, completion, closure, and safe mutation
  uses.
- **CP002 Planned Is Not Executed:** For every response with
  `verification_status: planned`, trust calibration must require execution or
  equivalent evidence before any passed-validation claim.
- **CP003 Direct-Read Scope Is Bounded:** Direct-read evidence may support
  precise claims only for the section or file represented by that evidence.
- **CP004 Capability Vocabulary Is Stable:** Adding trust calibration must not
  change meanings of capability level, evidence kind, freshness, analysis
  validity, or verification status.
- **CP005 Shared Policy Covers Public Surfaces:** Every public MCP resource or
  tool that returns a standard envelope must either have a trust calibration
  policy or an explicit documented exclusion.
- **CP006 Failure States Cannot Be Proof:** Error, warning, blocked, invalid,
  stale, degraded, and environment-failure responses must not be safe for proof
  or completion claims until the named missing evidence is supplied.
- **CP007 Additive Compatibility:** Trust calibration must be additive for
  existing response consumers unless the design explicitly selects a
  contract-version migration.

## Resolved Design Questions

### OQ001: Field Location

- **Why it matters:** The implementation can add trust calibration directly to
  `ResponseMetadata` or use a sibling shared structure. The wrong location
  could make every response heavier or create a second metadata contract.
- **Affected stage:** Design.
- **Resolution:** Resolved by `design.md` D001. Trust calibration is an
  optional `meta.trust` field on `ResponseMetadata`.
- **Blocking status:** Resolved for implementation planning.
- **Artifact destination:** `design.md`, `tasks.md`, and `traceability.md`.

### OQ002: Vocabulary Shape

- **Why it matters:** Values can be concise enums, short strings, or structured
  claim categories. Free-form strings are easier initially but harder to test.
- **Affected stage:** Design.
- **Resolution:** Resolved by `design.md` D002. Trust uses and verification
  requirements are enum-backed structured values.
- **Blocking status:** Resolved for implementation planning.
- **Artifact destination:** `design.md`, contract tests, `tasks.md`, and
  `traceability.md`.

### OQ003: Coverage Slice

- **Why it matters:** Covering every MCP surface in one implementation may be
  broad. Covering too few surfaces leaves the highest-risk outputs unchanged.
- **Affected stage:** Design and tasks.
- **Resolution:** Resolved by `design.md` D003. The implementation slice covers
  every public Workbench MCP resource or tool returning the standard envelope;
  non-framable transport startup failures and private/debug helpers are the
  only expected exclusions.
- **Blocking status:** Resolved for implementation planning.
- **Artifact destination:** `design.md`, `tasks.md`, and `traceability.md`.

## Success Criteria

- Requirements are accepted with no unresolved requirement-stage blockers.
- Design selects an additive trust calibration location and typed vocabulary.
- Major tool families have explicit calibration policies or documented
  exclusions.
- Contract and golden tests prove routing evidence is not proof and planned
  validation is not executed validation.
- Durable docs are promoted before closure.
